# 智慧实验室（Smart Lab Hub）产品设计方案

## 1. 产品定位

**智慧实验室**是一个面向教育科研、企业研发与检测机构的**远程实验室联邦平台**。运营方或教师在「中心控制台」统一接入多个物理实验室；用户选择目标实验室、下发实验指令；边缘侧执行并回传**结构化结果**与**实时/回放视频**，形成「选实验室 → 发指令 → 看过程 → 拿结果」的闭环。

| 维度 | 说明 |
|------|------|
| 核心价值 | 打破地域限制，复用稀缺设备与专家资源 |
| 典型用户 | 教师、学生、实验员、实验室管理员、平台运营 |
| 与竞品差异 | 以「多实验室联邦 + 指令编排 + 结果/视频双通道」为核心，而非单一设备的简单远控 |

### 1.1 建设意义

**「智慧」** 并非简单把摄像头接到网上，而是让分散在各校区、各楼宇的实验室成为**可调度、可协作、可留痕**的数字化资产。

| 意义维度 | 说明 |
|----------|------|
| **教育公平与普惠** | 偏远校区、分班教学的学生也能使用本部稀缺设备（示波器、通风橱、光学平台等），缩小因场地与机位不足造成的差距。 |
| **跨校区资源共建** | 多实验室统一接入、统一状态可见，避免重复建设；空闲机位可被全校共享，提高设备利用率。 |
| **实验过程可复现** | 指令、数据、视频与实验步骤绑定归档，便于教学复盘、评阅争议追溯与科研/检测存证。 |
| **安全可控的远程化** | 危化、高压等场景保留现场联锁与急停优先权；远程操作在剧本与审批约束下进行，降低误操作风险。 |
| **教学与运维一体化** | 教师编排实验课、学生按步骤操作；管理员巡检与健康监测同一平台完成，降低多套系统并存成本。 |

**一句话**：智慧实验室 = 用**联邦接入 + 标准指令 + 视频与数据双通道**，把「进实验室做实验」延伸到「在任何地点安全、规范地完成实验并留下完整记录」。

### 1.2 产品介绍（首页展示文案）

**标语**：面向国家级科研体系的实验资源统一接入、远程协同、过程监管、数据沉淀与安全治理平台。

**简介**：

面向科研任务的新型实验基础设施，统一连接实验室、设备与数据，支持远程调度、过程监控和结果回传，构建网络化、智能化、可追溯的实验基础设施。

远程实验过程受权限体系、流程规则、任务审批与安全策略约束，确保实验任务在合规边界内执行。现场急停、设备保护与安全联锁始终具备最高优先级，保障人员、设备、样本与实验环境安全。

整体来看，智慧实验室不是单一远程实验工具，而是面向国家级科研体系的实验资源统一接入、远程协同、过程监管、数据沉淀与安全治理平台。

## 2. 用户与场景

### 2.1 角色

| 角色 | 诉求 |
|------|------|
| **学生** | 远程完成指定实验，查看步骤视频与评分结果 |
| **教师** | 编排实验课、下发任务、批阅、课堂大屏监控多实验室状态 |
| **实验员（现场）** | 接收异常告警、必要时现场介入、维护设备 |
| **实验室管理员** | 注册实验室、配置设备清单、权限与配额 |
| **平台运营** | 多租户、计费、审计、SLA 监控 |

### 2.2 核心场景

1. **远程实验课**：教师创建「电路测量」任务 → 班级学生各自选择空闲的「电子学实验室 A」→ 下发「连接电路并测量 R1 电压」→ 实时看示波器画面 → 提交数据表与截图 → 系统自动判分。
2. **跨校区共享**：本部与分部各有一套化学实验室，分部学生通过平台使用本部通风橱做高危演示（教师审批 + 全程录像）。
3. **企业检测外协**：委托方下发检测脚本（采样、读数、报告模板），合作实验室执行并回传原始数据与过程视频存档。
4. **设备巡检**：管理员对全部实验室下发「开机自检」指令，汇总健康报告。

## 3. 产品功能架构

```
┌─────────────────────────────────────────────────────────────┐
│                    智慧实验室 · 中心平台                      │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│ 实验室目录   │ 指令与任务   │ 结果与媒体   │ 运营与治理         │
│ · 地图/列表  │ · 模板库     │ · 结构化数据 │ · 租户/权限        │
│ · 状态/排队  │ · 编排/审批  │ · 视频直播   │ · 审计日志         │
│ · 能力标签   │ · 定时/批量  │ · 录像回放   │ · 配额/计费        │
└─────────────┴─────────────┴─────────────┴───────────────────┘
                              │
                    安全接入层（VPN / mTLS / 边缘网关）
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   实验室 A              实验室 B              实验室 C
   边缘 Agent            边缘 Agent            边缘 Agent
   设备驱动 + 采集        设备驱动 + 采集        设备驱动 + 采集
```

### 3.1 实验室联邦（目录与连接）

- **实验室注册**：名称、地理位置、学科标签（物理/化学/生物/工科）、可容纳并发数、开放时段。
- **在线状态**：离线 / 空闲 / 占用中 / 维护中；心跳与最后同步时间。
- **能力描述**：支持的指令类型（如 `measure_voltage`、`start_reaction`）、绑定设备型号、视频机位列表。
- **预约与排队**：按时段预约；占用超时自动释放；优先级队列（教学 > 巡检）。

### 3.2 指令系统（核心）

指令是平台与边缘之间的**标准契约**，建议分层：

| 层级 | 示例 | 说明 |
|------|------|------|
| **原子指令** | `device.power_on`、`camera.switch` | 单设备单动作 |
| **实验步骤** | `step.connect_circuit` | 可带参数、超时、前置条件 |
| **实验剧本** | 「欧姆定律验证」 | 多步骤 DAG，支持分支与人工确认点 |
| **批处理** | 对 10 个实验室下发 `self_test` | 运营/巡检场景 |

**指令生命周期**：`草稿 → 已下发 → 已接收 → 执行中 → 成功/失败/超时 → 结果已归档`

关键字段（概念模型）：

```json
{
  "commandId": "cmd_20260520_001",
  "labId": "lab_electronics_a",
  "type": "experiment.run_step",
  "payload": { "stepId": "measure_r1", "params": { "channel": 1 } },
  "priority": "normal",
  "timeoutSec": 300,
  "requireVideo": true,
  "callbackUrl": "https://platform/api/v1/commands/callback"
}
```

### 3.3 结果与视频

**结构化结果**

- 数值、表格、JSON、文件（CSV/PDF/图片）、设备原始 trace。
- 与指令 `commandId` 强绑定；支持版本与补传。

**视频**

- **直播**：WebRTC / HLS 低延迟多机位（全景 + 特写仪器屏幕）。
- **录像**：指令开始到结束自动关联存档；水印（实验室、用户、时间）。
- **同步**：结果时间轴上打点标记「关键帧」，点击结果可跳转对应视频片段。

### 3.4 控制台与终端体验

**Web 控制台（教师/管理员）**

- 实验室地图/列表筛选；一键进入「监控墙」（多路视频宫格）。
- 剧本编辑器：拖拽步骤、参数表单、模拟下发（沙箱实验室）。
- 课堂模式：统一开始、锁定学生只能访问指定实验室、大屏投屏。

**学生端（Web / 小程序 / App）**

- 我的任务 → 选择实验室 → 分步引导 → 实时画面 + 操作面板（仅允许剧本内动作）。
- 提交实验报告（自动填充已采集数据）。

**现场边缘 UI（可选）**

- 本地大屏显示「远程占用中」、急停、物理安全联锁状态。

## 4. 技术架构

### 4.1 逻辑分层

```
[ 用户层 ]  Web / App / 大屏
      ↓ HTTPS / WebSocket
[ 业务层 ]  API Gateway · 用户/权限 · 任务编排 · 消息 · 媒资 · 报表
      ↓ MQTT / gRPC (mTLS)
[ 接入层 ]  实验室网关（每站点 1 台）· 协议适配 · 离线队列
      ↓ 本地总线
[ 边缘层 ]  Lab Agent · 设备驱动插件 · 视频采集 · 安全联锁
      ↓
[ 物理层 ]  仪器、PLC、摄像头、传感器
```

### 4.2 推荐技术选型（可替换）

| 组件 | 建议 | 理由 |
|------|------|------|
| 中心 API | Go / Java + REST + WebSocket | 稳定、易对接学校 IT |
| 指令通道 | MQTT 5（QoS1）+ 指令幂等键 | 弱网、断线重连友好 |
| 视频 | SRS / MediaMTX + WebRTC；录像落对象存储 | 开源、可私有化 |
| 边缘 Agent | Python/Go + 插件 SO/DLL 驱动 | 对接仪器 SDK 灵活 |
| 数据库 | PostgreSQL（业务）+ Redis（会话/锁）+ MinIO（文件） | 成熟组合 |
| 检索/审计 | Elasticsearch 或 PG 全文 | 指令与日志追溯 |

### 4.3 关键数据流

**下发指令**

1. 用户在控制台创建任务 → 平台校验权限与实验室状态。
2. 生成 `commandId`，写入 Outbox，经 MQTT 发往 `lab/{labId}/commands`。
3. 边缘 Agent 校验签名与配额 → 入本地队列 → 驱动执行。
4. Agent 上报 `ACK` → `PROGRESS` → `RESULT` / `ERROR`。

**回传结果与视频**

- 小结果：MQTT `lab/{labId}/results` 或 HTTPS 回调。
- 大文件/录像：预签名 URL 直传对象存储，元数据写回平台。
- 直播：边缘推 RTMP/WebRTC 到媒体服务；平台签发限时观看 Token。

### 4.4 多实验室隔离

- **逻辑隔离**：`tenantId` + `labId`；指令与媒资命名空间分离。
- **网络隔离**：每实验室独立 VPN 或反向隧道（frp/Tailscale），中心不直接暴露设备 IP。
- **资源隔离**：单实验室最大并发指令数、带宽上限、存储配额。

## 5. 安全与合规

| 类别 | 措施 |
|------|------|
| 身份 | OIDC/校企统一认证；RBAC + 实验室级 ACL |
| 通信 | 全链路 TLS；边缘证书轮换；指令 HMAC 签名 |
| 安全联锁 | 危化/高压实验强制「本地急停优先」；远程指令不得绕过硬件互锁 |
| 审计 | 谁、何时、对哪间实验室、发了什么、视频是否被下载 |
| 隐私 | 视频脱敏区域（人脸、铭牌）； retention 策略可配置 |
| 合规 | 教育数据分级；跨境传输评估（若多地域部署） |

## 6. 非功能需求

| 指标 | 目标（可调整） |
|------|----------------|
| 指令下发延迟 | P95 < 2s（国内专线/VPN） |
| 视频延迟 | 直播端到端 < 1s（WebRTC 局域网）/ < 3s（跨网 HLS） |
| 可用性 | 中心 99.9%；单实验室离线不影响其他实验室 |
| 离线 | 边缘队列缓存指令 24h；恢复后顺序执行或人工仲裁 |
| 扩展 | 单集群 ≥ 500 实验室；横向扩展媒体与 MQTT 集群 |

## 8. MVP 与路线图

### Phase 1 — MVP（8～12 周）

- [ ] 实验室注册、在线状态、基础目录

### Phase 3 — 生态

- [ ] 开放指令 SDK、第三方设备插件市场
- [ ] 多租户计费、数据看板
- [ ] AI 辅助（异常检测、实验步骤纠错、视频摘要）

## 10. 边缘 Agent 接口约定（摘要）

**上行主题**

- `lab/{labId}/status` — 心跳、资源占用
- `lab/{labId}/commands/{commandId}/ack|progress|result|error`
- `lab/{labId}/events` — 告警、急停、维护模式

**下行主题**

- `lab/{labId}/commands` — 新指令
- `lab/{labId}/config` — 配置热更新

**HTTP 补充**

- `POST /edge/v1/upload` — 大文件与录像元数据
- `GET /edge/v1/health` — 探活

## 11. 成功指标（KPI）

| 指标 | 说明 |
|------|------|
| 实验室月活连接率 | 注册实验室中 ≥80% 月在线 |
| 指令成功率 | 不含用户取消，≥95% |
| 单次实验完成时长 | 相对现场实验的耗时比 |
| 视频卡顿率 | 会话级 <5% |
| 安全事故 | 远程误操作导致的事故为零（联锁达标） |

## 12. 风险与对策

| 风险 | 对策 |
|------|------|
| 仪器协议各异 | 插件化驱动 + 先做 2～3 个标准学科示范实验室 |
| 网络不稳定 | MQTT 持久会话 + 边缘队列 + 结果补传 |
| 责任边界不清 | 剧本审批、录像存证、现场急停权责写入使用协议 |
| 视频成本高 | 按需直播、结束即停录；分级存储与过期删除 |

## 13. 下一步建议

1. **确定首批试点实验室**（学科、设备清单、网络条件）。
2. **冻结 MVP 指令集**（5～10 条原子指令 + 1 套完整示范剧本）。
3. **搭建最小闭环**：中心平台 + 单边缘 Agent + 1 路视频 + 模拟设备。
4. 在本仓库按 `docs/` 方案拆分服务：`platform-api`、`edge-agent`、`media`、`web-console`。
